## Problem Statement

The schedule page shows a hardcoded Kanban board with static dates, mock activity data, and no persistence. Every page refresh resets all state. There is no concept of the actual current week, no way to distinguish activity types (climbing session vs conditioning vs mobility vs warm-up), and no backend to store what the user has planned. The app cannot be used as a real training tool until scheduling is grounded in real data.

## Solution

Replace the static schedule page with a live, persisted experience backed by Supabase. The user logs in once via magic link and their weekly schedule persists across sessions. Days display the actual current calendar week. Activities are typed — climbing sessions, conditioning, mobility, and warm-up — each with appropriate fields. The user can add activities through a two-step modal (pick category → fill type-specific form), delete activities, drag to reorder within a day or move between days, and navigate freely to past and future weeks.

## User Stories

1. As a climber, I want to see the actual current week in the schedule, so that I know I am planning for real dates.
2. As a climber, I want to navigate to past weeks, so that I can review what I trained previously.
3. As a climber, I want to navigate to future weeks, so that I can plan training blocks ahead of time.
4. As a climber, I want a "This week" shortcut, so that I can return to the current week instantly after navigating away.
5. As a climber, I want days with no activities to appear as rest days, so that I can see my recovery at a glance without doing any extra work.
6. As a climber, I want to log in with a magic link sent to my email, so that I do not have to manage a password.
7. As a climber, I want my schedule to persist after closing the browser, so that I do not have to re-enter my plan every session.
8. As a climber, I want to add an activity to any day in the schedule, so that I can build my weekly training plan.
9. As a climber, I want to choose the activity category first (Climbing Session, Conditioning, Mobility, Warm Up) when adding an activity, so that I see only the relevant fields for that type.
10. As a climber, I want to add a climbing session with an intent linked to the skill tree, so that my session has a clear training focus.
11. As a climber, I want to add a climbing session with a duration in minutes, so that I can plan how long I will be at the wall.
12. As a climber, I want to pick a conditioning activity from a curated suggested list, so that I do not have to type common exercises from scratch.
13. As a climber, I want to type a custom conditioning activity, so that I can log anything not on the suggested list.
14. As a climber, I want to pick a mobility activity from a curated suggested list, so that I do not have to type common stretches from scratch.
15. As a climber, I want to type a custom mobility activity, so that I can log anything not on the suggested list.
16. As a climber, I want to add a warm-up block (General Warm Up) that shows its constituent exercises, so that I know exactly what the block contains.
17. As a climber, I want to see the exercises within a template activity (e.g. General Warm Up), so that I do not have to remember what it includes.
18. As a climber, I want to drag an activity to reorder it within a day, so that my day is structured the way I want (e.g. Warm Up before Climbing Session).
19. As a climber, I want to drag an activity from one day to another, so that I can reschedule without deleting and re-adding.
20. As a climber, I want the order I set for activities to persist after refreshing, so that I do not have to re-sort my day each time.
21. As a climber, I want to delete an activity from a day, so that I can remove something I no longer plan to do.
22. As a climber, I want the activity card to display the relevant details (type, intent for climbing, title for others), so that I can read my plan at a glance.

## Implementation Decisions

### Supabase Setup
- New Supabase project created via dashboard (no CLI for initial schema setup).
- Supabase JS client initialised once and exported from a shared module.
- Row-level security enabled on the activities table so users can only read and write their own rows.

### Authentication
- Supabase Auth with magic link (OTP via email).
- Minimal auth UI: single email input and a send-link button.
- Auth state managed via a React context/hook wrapping the Supabase auth listener.
- The rest of the app is only rendered once a valid session exists.

### Database Schema
Single `activities` table:
- `id` — uuid primary key
- `user_id` — uuid, foreign key to auth.users, not null
- `scheduled_date` — date, not null (ISO 8601)
- `type` — text, not null; one of `climbing`, `conditioning`, `mobility`, `warmup`
- `title` — text, not null; for non-climbing activities this is the activity name; for climbing it is derived from intent
- `intent_node_id` — text, nullable; climbing sessions only; references a skill tree node ID
- `duration_minutes` — integer, nullable; climbing sessions only
- `order` — integer, not null; controls display order within a day
- `created_at` — timestamptz, default now()

No separate week, day, or training-day rows. Rest days are implicit: any calendar date with zero activities is a rest day.

### Activity Templates (hardcoded in app)
Suggested lists and exercise breakdowns are hardcoded in the codebase — no database table. Each template defines a name, category, and an optional ordered list of exercises.

Conditioning suggestions: Weighted Pull Ups, Shoulder Press, Bench Press, Deadlift, Seated Shoulder Rotation.

Mobility suggestions: Ankle Flexibility (exercises: Calf Stretch, Donkey Calf Raise, Fishermans - Passive), Pancake Stretch, Revolver Stretch, Cossack Squats, Pigeon Pose.

Warm-up templates: General Warm Up (exercises: Leg Swings, Scapular Push Ups, Cossack Squats, Face Pulls, Split Squats).

### Schedule Data Layer
A dedicated data-access module wraps all Supabase calls for activities:
- Fetch activities for a date range (a full ISO week)
- Insert a new activity
- Delete an activity by ID
- Update `order` on a set of activities (for drag reorder within a day)
- Update `scheduled_date` and `order` on an activity (for drag between days)

### SchedulePage
- Derives the current ISO week on mount; week navigation increments/decrements by 7 days.
- Fetches activities from Supabase for the displayed week on mount and on week change.
- Optimistic UI: updates local state immediately on add/delete/drag, rolls back on Supabase error.
- Drag-and-drop (existing @hello-pangea/dnd) persists order and date changes to Supabase.

### AddActivityModal (redesigned)
Two-step flow:
1. Category picker — four large buttons: Climbing Session, Conditioning, Mobility, Warm Up. Back button returns here from step 2.
2. Type-specific form:
   - Climbing Session: skill tree intent picker + duration (minutes) input.
   - Conditioning / Mobility: scrollable suggested list + free-text input for custom entries.
   - Warm Up: single-tap to add the General Warm Up block (shows its exercise list inline).

### ActivityCard
Existing component extended with a delete button (trash icon). Confirmation prompt before deletion.

## Testing Decisions

Good tests verify observable behaviour through the module's public interface, not internal implementation details. A test that breaks when you rename a private function is not a good test.

Modules to test:

- **Activity templates module** — pure data: verify each template has required fields, exercises are correctly associated, category groupings are accurate. No mocks needed.
- **Week date utilities** — pure functions: given a reference date, assert the correct Monday-Sunday range is returned; assert week navigation increments correctly; assert "this week" snaps back to today's week.
- **Activities data layer** — integration tests against a Supabase test project or local emulator: verify insert, fetch-by-range, delete, and order-update produce the expected database state.

The existing `suggestions.ts` utility already has test coverage patterns in the codebase that serve as prior art for pure-function tests.

## Out of Scope

- Activity editing (only delete is supported at launch).
- Logging/completion tracking (marking activities as done, sets/reps, etc.).
- Exercise-level data stored in the database (exercises remain hardcoded template data).
- Additional activity categories beyond the four defined (climbing, conditioning, mobility, warmup).
- Exact skill tree intent picker granularity (column exists; UI depth is a follow-up decision).
- Multi-user features, sharing, or social functionality.
- Offline support or conflict resolution.
- Activity library page (remains a placeholder).
- OAuth providers (only magic link at launch).
- Supabase CLI or migration tooling (dashboard-based schema management for now).

## Further Notes

The `order` column approach for drag-reorder requires updating multiple rows on every reorder within a day. For the current single-user scope this is acceptable. A fractional indexing scheme could reduce writes in future but is unnecessary now.

The activity template data (suggested lists, exercise breakdowns) is intentionally hardcoded to avoid premature infrastructure. The set of templates is expected to evolve as the user tests the app; once it stabilises, migrating to a database-backed `activity_templates` table is a clean follow-up.

The skill tree intent link for climbing sessions stores any node ID. The decision of whether to restrict the picker to top-level categories or allow deeper node selection is deferred and can be made when building the intent picker UI.
